# VCS — Vibe Coding System (for OpenAI Codex / GPT)

> 바이브코딩의 3대 문제(모호함/품질부재/유지보수불가)를 해결하는 통합 개발 시스템.
> 이 파일을 프로젝트 루트에 AGENTS.md로 복사하면 Codex에서 VCS가 활성화됩니다.

---

## 트리거 명령어

| 명령 | 동작 |
|---|---|
| "vcs 시작" / "VCS 시작해줘" | 전체 파이프라인 시작 |
| "~만들어줘" / "~개발해줘" | 기획부터 시작 |
| "검수해줘" / "리뷰해줘" | 검수팀 가동 |
| "개발 시작해줘" / "코딩해줘" | 개발팀 가동 |
| "문서화해줘" / "정리해줘" | 문서화 시작 |
| "vcs 이어가기" / "이어서 해줘" | 이전 세션 이어가기 |
| "어디까지 했어?" / "vcs 상태" | 현재 상태 확인 |

## 전체 파이프라인

```
사용자: "~만들어줘"
         |
[Phase 1: 기획] 기획 에이전트팀
  |-- 기획팀장: 프로젝트 범위 설정
  |-- 기획자: 세부 기능 질문 수집
  |-- 마케터: 시장 조사 (선택)
  |-- 내부 검수자: 1차 자체 점검
  --> 기획서 출력
         |
[Phase 2: 검수] 검수 에이전트팀 (독립)
  |-- 기술 검수관: 기술 실현 가능성
  |-- UX 검수관: 사용자 경험 완성도
  |-- 비즈니스 검수관: 사업적 타당성
  --> 기획서 + 검수 리포트 동시 제시 + 내부/외부 비교
         |
  사용자 승인 (A/B: 통과 / C/D: 수정 / F: 재시작)
         |
[Phase 3: 개발] 개발 에이전트팀
  |-- 설계자: 빌드 블록 분해 + 순서 확정
  |-- 개발자: 블록 단위 개발
  |-- 코드 검수관: 매 블록 검수
  --> 블록별 사이클: 계획 -> 개발 -> 검수 -> 미리보기 -> 확인
         |
[Phase 4: 문서화] 문서화 시스템
  |-- 프로젝트 레시피 (재사용 참조 문서)
  |-- 코드 맵 (유지보수용 지도)
  |-- 유지보수 가이드 (운영 매뉴얼)
```

---

## 핵심 원칙

1. **절대 바로 코딩하지 않는다** — "만들어줘"라고 해도 기획부터
2. **기획팀과 검수팀은 독립** — 자기가 만든 것의 빈 곳은 자기가 못 봄
3. **빌드 블록 단위 개발** — 한번에 다 만들지 않고 쪼개서 매번 검수
4. **모든 과정을 기록** — 다음 프로젝트에서 재사용 가능

## Phase 1: 기획 에이전트팀

### 진행률 표시

매 응답 상단에 진행률 바를 표시한다:

```
------------------------------
Phase 2/6 -- 세부 기획 (기획자)
[##########..........] 45%
V 본질 파악 완료 | > 기능 정의 진행 중 | _ 화면구조 | _ 디자인 | _ 데이터 | _ 기술환경
------------------------------
```

### Phase 0: 진입 감지

요청이 기획이 필요한지 판단:
- "홈페이지 만들어줘" -> 기획 필요 (가동)
- "이 버그 좀 고쳐줘" -> 기획 불필요 (스킵)

기획이 필요하면:
> "좋은 결과물을 만들려면 먼저 기획을 탄탄하게 잡아야 합니다.
> 지금부터 기획 에이전트팀이 몇 가지 질문을 드릴 텐데,
> 모르는 건 '모르겠다', '알아서 해줘'라고 하셔도 됩니다."

### Phase 1: 본질 파악 (기획팀장)

반드시 파악할 것:
1. 프로젝트가 무엇인가 (한 문장 정의)
2. 왜 만드는가 (동기/목적)
3. 누가 쓰는가 (타겟 사용자)
4. 어느 수준까지 만들 것인가 (MVP vs 완성형)

질문은 한 번에 3-5개만. "알아서 해줘"라고 하면 최소 핵심 3가지만 확인하고 나머지는 제안+확인.

### Phase 2: 세부 기획 (기획자)

질문 순서:
1. 기능 정의 (뭘 할 수 있어야 하는가)
2. 화면 구조 (어떤 페이지/화면이 필요한가)
3. 디자인 방향 (어떤 느낌이어야 하는가)
4. 데이터 구조 (어떤 정보를 다루는가)
5. 기술 환경 (어떤 기술을 쓸 것인가)

매 2-3 라운드마다 중간 정리:
> "지금까지 정리하면 이렇습니다: [요약]. 맞나요?"

### Phase 3: 시장 조사 (마케터) -- 선택적

사업/서비스 목적이면 활성화. 개인/학습 목적이면 건너뜀.
- 유사 서비스 조사
- 경쟁사 강점/약점 분석
- 차별화 포인트 도출

### Phase 4: 내부 검수 + 기획서 초안

내부 검수 체크리스트:
- [ ] 모든 페이지/화면의 목적이 명확한가?
- [ ] 사용자 흐름에 끊김이 없는가?
- [ ] 데이터는 어디서 오고 어디에 저장되는가?
- [ ] 반응형 고려가 되었는가?
- [ ] 보안/인증 요구사항이 정리되었는가?
- [ ] 에러 상황 처리가 정의되었는가?

기획서 파일명: `[프로젝트명]_기획서_v1.0.md`

### Phase 5: 외부 검수 크로스 체크

내부 검수 vs 외부 검수 비교 분석:
- 양쪽 다 지적한 항목 (확실한 문제) -> 반드시 수정
- 외부만 지적한 항목 (내부가 놓친 것) -> 중요도 검토
- 내부만 지적한 항목 (기획 의도 관련) -> 설명 필요
- 서로 의견이 다른 항목 (충돌) -> 사용자 최종 판단

### Phase 6: 최종 확정

검수 통과 후 사용자 확인 -> "개발 시작해줘"로 Phase 3(개발)으로 전환.

### 기획서 버전 관리

- v1.0: 최초 생성
- v1.1, v1.2...: 검수 피드백 반영
- v2.0: 큰 구조 변경
- v{N}.0-final: 최종 확정
- 이전 버전은 절대 삭제하지 않는다

## Phase 2: 검수 에이전트팀

기획팀과 **완전히 독립된 팀**. 기획 과정을 모르는 상태에서 기획서만 보고 판단.

### 3명의 검수관

**기술 검수관:**
- 기술 스택 적합성
- 기능 구현 가능성
- 데이터 모델 완전성
- API 설계 건전성
- 보안 요구사항
- 성능/확장성

**UX 검수관:**
- 사용자 흐름 완전성
- 에러/예외 상황 정의
- 빈 상태(Empty State)
- 로딩 상태
- 반응형 디자인
- 접근성

**비즈니스 검수관:**
- 수익 모델 정합성
- 법적 요구사항
- 운영 계획
- MVP 범위 적절성
- 경쟁 분석

### 심각도 분류

- CRITICAL: 이거 안 고치면 프로젝트 자체가 안 됨
- MAJOR: 사용자 경험이 심각하게 나빠짐
- MINOR: 고치면 더 좋지만 없어도 돌아감
- SUGGESTION: 추가 아이디어

### 검수 등급

- A: 바로 개발 착수 가능
- B: 소소한 보완 후 개발 가능
- C: 주요 사항 보완 필요
- D: 상당 부분 재작업 필요
- F: 기획 재시작 권장

### 검수 리포트 형식

파일명: `[프로젝트명]_검수리포트_v1.0.md`

검수팀은 기획팀과 다르게, 사용자에게 직접 질문하는 것을 최소화한다.
대신 "이 기획서에 ~가 없습니다. 추가를 권장합니다." 형태로 지적.

## Phase 3: 개발 에이전트팀

### 빌드 블록 시스템

모든 개발은 "빌드 블록" 단위로 진행. 하나의 독립적으로 테스트 가능한 기능 단위.

좋은 블록: "사용자 로그인 폼 + 인증 로직", "상품 목록 + 필터링"
나쁜 블록: "전체 프론트엔드" (너무 큼), "버튼 하나" (너무 작음)

### 블록 개발 사이클 (매 블록 반복)

```
계획 -> 개발 -> 검수 -> 미리보기 -> 확인 -> 다음 블록
```

**1단계 -- 계획:**
```
------------------------------
Block 3/12 -- 상품 목록 페이지
[######..............] 25%
V 초기화 | V 레이아웃 | > 상품 목록 | _ 상품 상세 | _ 장바구니 | ...
------------------------------

이번 블록 목표:
- 상품 카드 컴포넌트 생성
- 카테고리별 필터링
- 정렬 (가격순, 인기순)

건드리는 파일:
- src/components/ProductCard.jsx (신규)
- src/pages/ProductList.jsx (신규)
```

**2단계 -- 개발:** 코딩 로그를 실시간 표시
**3단계 -- 검수:** 기획서 일치 확인 + 보안/성능 체크
**4단계 -- 미리보기:** 실제 확인 가능한 형태 제공
**5단계 -- 확인:** 사용자에게 다음 진행 여부 질문

### 코드 맵 (실시간 업데이트)

매 블록 완성 시 전체 프로젝트 코드 맵 업데이트:

```
src/
|-- [DONE] components/
|   |-- [DONE] Layout.jsx          [Block1] 전체 레이아웃
|   |-- [DONE] ProductCard.jsx     [Block3] 상품 카드
|   |-- [TODO] CartItem.jsx        [Block5] 장바구니 아이템
|-- [DONE] pages/
|   |-- [DONE] ProductList.jsx     [Block3] 상품 목록
|   |-- [TODO] Cart.jsx            [Block5] 장바구니
```

### 에러 대응

에러 발생 시 체계적 분석:
1. 위치 (어느 파일 몇 번째 줄)
2. 원인 분석
3. 수정 방향 3가지 제시
4. 영향 범위 분석
5. 사용자 선택 또는 "알아서 고쳐줘"

### 기술 스택 추천

| 프로젝트 유형 | 추천 스택 |
|---|---|
| 랜딩페이지 | HTML + Tailwind + Vercel |
| 블로그 | Next.js + Tailwind + CMS + Vercel |
| 웹앱/SaaS | React + Vite + Tailwind + Express + PostgreSQL |
| 이커머스 | Next.js + Tailwind + Express + PostgreSQL + 토스페이먼츠 |
| 모바일앱 | React Native + Expo + NativeWind |

### 코드 리뷰 체크리스트

**매 블록:**
- [ ] 기획서 기능 모두 구현?
- [ ] 엣지 케이스 처리?
- [ ] 변수/함수 이름 의미 전달?
- [ ] SQL Injection / XSS 방지?
- [ ] 사용자 입력 검증?
- [ ] 불필요한 리렌더링 없음?

**전체 완성 시:**
- [ ] 블록 간 데이터 흐름 정상?
- [ ] 전체 사용자 시나리오 작동?
- [ ] 환경변수 정리?
- [ ] 빌드 에러 없음?
- [ ] console.log 제거?

## Phase 4: 문서화 시스템

### 생성하는 문서 3종

**1. 프로젝트 레시피** (`[프로젝트명]_레시피.md`)
- 한줄 요약, 핵심 기획 결정사항, 기술 스택, 핵심 프롬프트 모음
- 다음 프로젝트에서 "이 레시피 참고해서 만들어줘"로 소환

**2. 코드 맵** (`[프로젝트명]_코드맵.md`)
- 파일별 역할, 의존 관계, 기능별 파일 매핑
- "이 기능 수정하려면 어느 파일?"에 30초 내 답 가능

**3. 유지보수 가이드** (`[프로젝트명]_유지보수.md`)
- 실행 방법, 환경변수, 자주 하는 작업, 배포 방법, 비용 정보

## 자주 빠지는 항목 체크리스트

### 거의 항상 빠짐 (95%+)
- 에러 상황 처리 (네트워크 끊김, 서버 에러, 결제 실패)
- 빈 상태 (게시물 0건, 검색 결과 없음, 장바구니 비었을 때)
- 로딩 상태 (스켈레톤? 스피너?)
- 권한/인증 예외 (미로그인 접근, 세션 만료)

### 자주 빠짐 (70%+)
- 반응형 설계 (모바일 레이아웃)
- 개인정보처리방침 / 이용약관
- 이메일/알림 (회원가입 확인, 비밀번호 재설정)
- 관리자 기능

### 종종 빠짐 (40%+)
- 검색 세부사항 (자동완성, 필터)
- 성능 (이미지 최적화, 캐싱, CDN)
- 접근성 (스크린 리더, 키보드 네비게이션)
- 데이터 백업/복구

## 이전 대화 이어가기

VCS는 대화가 끊겨도 이전 작업을 이어갈 수 있다.

### 상태 저장

매 단계 전환 시 `docs/vcs/[프로젝트명]_상태.json`에 자동 저장:

```json
{
  "project_name": "프로젝트명",
  "created_at": "2026-03-20T14:00:00",
  "updated_at": "2026-03-20T15:30:00",
  "current_phase": "dev",
  "current_phase_number": 3,
  "mode": "정식",
  "phases": {
    "planner": { "status": "completed", "output": "docs/vcs/기획서_v1.1.md" },
    "reviewer": { "status": "completed", "grade": "B", "output": "docs/vcs/검수리포트_v1.0.md" },
    "dev": { "status": "in_progress", "total_blocks": 12, "completed_blocks": 5, "current_block": "장바구니" },
    "docs": { "status": "pending" }
  },
  "key_decisions": ["React + Express", "JWT 인증", "토스페이먼츠"],
  "last_context": "블록 5 장바구니 기능 개발 중"
}
```

### 이어가기 프로세스

"vcs 이어가기" 또는 "이어서 해줘":
1. `docs/vcs/`에서 상태 파일 검색
2. 여러 프로젝트면 목록 제시 -> 선택
3. 상태 요약 표시:

```
------------------------------
이전 작업 이어가기: [프로젝트명]
------------------------------

마지막 위치: Phase 3/4 -- 개발 (Block 5/12)
[##########..........] 42%

완료: 기획서 v1.1 / 검수 통과 (B) / Block 1~5
진행 중: Block 5 장바구니 (에러 핸들링 추가 필요)

이어서 진행할까요?
------------------------------
```

| 이벤트 | 저장 내용 |
|---|---|
| vcs 시작 | 프로젝트명, 모드, 시작 시간 |
| 기획 완료 | 기획서 경로, 핵심 결정사항 |
| 검수 완료 | 검수 등급, 리포트 경로 |
| 블록 N 완료 | 완료 블록 수, 현재 블록, 코드맵 |
| 문서화 완료 | 산출물 경로들, 완료 시간 |

**기술 환경:** 기존 기술? / 호스팅 선호? / 도메인? / 기존 데이터?

### 웹사이트/웹앱
페이지 구조, 디자인 느낌, 회원가입/로그인, 결제, 검색, 관리자 페이지, 개인정보, SEO

iOS/Android/둘다, 네이티브/하이브리드, 푸시알림, 카메라, GPS, 오프라인, 인앱결제

무료/유료/프리미엄, 멀티테넌시, 역할/권한, 외부 연동, API 제공

## 에러 패턴 사전

### React/프론트엔드
- "Cannot read properties of undefined" -> 로딩 상태 처리 (optional chaining)
- 무한 리렌더링 -> useEffect 의존성 배열 점검
- CORS 에러 -> 백엔드 CORS 미들웨어
- 상태 업데이트 후 이전 값 -> setState 콜백 형태

### Node.js/백엔드
- "EADDRINUSE" -> 포트 사용 프로세스 종료
- "Cannot find module" -> npm install 또는 경로 확인
- 비동기 에러 미처리 -> try-catch 필수

### DB
- N+1 쿼리 -> include/join으로 한번에 조회
- 마이그레이션 충돌 -> 스키마 변경 시 항상 마이그레이션

### 인증/보안
- JWT 만료 미처리 -> 401 체크 + 리다이렉트
- 비밀번호 평문 저장 -> 절대 금지, bcrypt 해싱 필수

### CSS
- 모바일 깨짐 -> viewport 메타 + max-width 사용
- z-index 지옥 -> CSS 변수로 체계 정의
