# 바이브코딩 문서화/유지보수 시스템

## 이 시스템의 존재 이유

바이브코딩의 세 번째 문제는 **"만들었는데 유지보수를 못 한다"**이다.

AI가 코딩했는데, 무슨 기술로 어떻게 만들었는지 바이브코더가 모른다.
그래서 조금만 수정하려 해도 전체를 다시 만들거나, 다른 AI에게 처음부터 다시 시킨다.

또 하나의 문제: **"어떤 프롬프트가 이 결과를 만들었는지 모른다."**
다음에 비슷한 프로젝트를 할 때 처음부터 다시 시작해야 한다.

이 시스템이 해결하는 것:
1. **프롬프트 기록**: 어떤 지시가 이 결과를 만들었는지 기록
2. **코드 맵**: 어떤 파일이 어떤 역할인지 한눈에 파악
3. **기술 문서**: 무슨 기술로 왜 이렇게 만들었는지 설명
4. **참조 문서**: 다음 프로젝트에서 소환해서 쓸 수 있는 md 파일

## 생성하는 문서 3종

### 1. 프로젝트 레시피 (Project Recipe)
**파일명:** `[프로젝트명]_레시피.md`

이 프로젝트를 처음부터 다시 만들려면 어떻게 해야 하는지의 "조리법".
다음 프로젝트에서 "이 문서 참고해서 비슷한 걸 만들어줘"라고 소환하는 용도.

```markdown
# [프로젝트명] 프로젝트 레시피

## 한줄 요약
> [이 프로젝트가 무엇인지 한 문장]

## 프로젝트 유형
[웹앱 / 모바일 / SaaS / 이커머스 / ...]

## 핵심 기획 결정사항
기획 과정에서 내린 핵심 결정들. 다음에 비슷한 프로젝트 할 때
같은 질문이 나오면 이걸 참고하면 된다.

| 결정 사항 | 선택한 것 | 왜 |
|-----------|----------|-----|
| 인증 방식 | JWT + 소셜로그인 | 사용자 편의성 우선 |
| DB 선택 | PostgreSQL | 관계형 데이터 + 무료 |
| 결제 연동 | 토스페이먼츠 | 한국 결제 최적화 |
| ...       | ...     | ... |

## 기술 스택
| 영역 | 기술 | 버전 | 선택 이유 |
|------|------|------|----------|
| 프론트엔드 | React | 18.x | ... |
| 백엔드 | Express | 4.x | ... |
| ...    | ...    | ... | ... |

## 핵심 프롬프트 모음
이 프로젝트를 만들 때 사용한 주요 프롬프트를 단계별로 정리.
다음에 비슷한 프로젝트를 할 때 이 프롬프트를 기반으로 지시하면 된다.

### 기획 단계 프롬프트
```
[실제 사용된 기획 요청 프롬프트]
```
→ 결과: [어떤 기획서가 나왔는지 요약]

### 개발 단계 핵심 프롬프트
```
[가장 효과적이었던 개발 프롬프트 1]
```
→ 결과: [어떤 코드가 나왔는지 요약]

```
[핵심 프롬프트 2]
```
→ 결과: [요약]

### 디버깅/수정 프롬프트
```
[에러 수정 시 사용한 효과적인 프롬프트]
```
→ 결과: [어떻게 해결되었는지]

## 프로젝트 구조 요약
```
[폴더 트리]
```

## 다음 프로젝트에서 이 레시피 사용법
1. 이 파일을 Claude에게 보여준다
2. "이 레시피를 참고해서 [새 프로젝트 설명]을 만들어줘"
3. Claude가 기술 스택, 구조, 패턴을 이 레시피 기반으로 적용
4. 필요한 부분만 수정하면서 진행
```

### 2. 코드 맵 (Code Map)
**파일명:** `[프로젝트명]_코드맵.md`

프로젝트의 모든 파일이 각각 뭘 하는지, 서로 어떻게 연결되는지의 지도.
유지보수할 때 "이 기능 수정하려면 어느 파일을 건드려야 하는가"를 바로 알 수 있게.

```markdown
# [프로젝트명] 코드 맵

## 전체 구조
```
src/
├── components/     [UI 컴포넌트]
├── pages/          [페이지]
├── api/            [API 통신]
├── hooks/          [커스텀 훅]
├── utils/          [유틸리티]
├── styles/         [스타일]
└── config/         [설정]
```

## 파일별 역할

### 🟠 핵심 파일 (건드리면 영향 큰 것)
| 파일 | 역할 | 의존하는 파일 | 이 파일에 의존하는 파일 |
|------|------|-------------|---------------------|
| src/api/auth.js | 인증 API 호출 | config/api.js | pages/Login, hooks/useAuth |

### 🔵 기본 파일
| 파일 | 역할 | 의존 관계 |
|------|------|----------|

### ⚪ 부가 파일
| 파일 | 역할 | 의존 관계 |
|------|------|----------|

## 기능별 파일 매핑

특정 기능을 수정하고 싶을 때 어느 파일을 봐야 하는지:

| 기능 | 관련 파일들 | 핵심 파일 |
|------|-----------|----------|
| 로그인 | Login.jsx, auth.js, useAuth.js | auth.js |
| 상품 목록 | ProductList.jsx, products.js, ProductCard.jsx | products.js |
| 결제 | Checkout.jsx, payment.js, Cart.jsx | payment.js |

## 데이터 흐름

주요 데이터가 어디서 시작해서 어디로 흐르는지:

```
사용자 입력 → 컴포넌트(상태) → API 함수 → 서버 → DB
                                           ↓
사용자 화면 ← 컴포넌트(렌더) ← 상태 업데이트 ← 응답
```

## 수정 가이드

자주 있는 수정 요청별 가이드:

### "새 페이지를 추가하고 싶다"
1. `src/pages/` 에 새 파일 생성
2. `src/App.jsx`의 라우터에 경로 추가
3. 네비게이션에 링크 추가

### "API를 추가하고 싶다"
1. `src/api/` 에 새 API 함수 생성
2. 백엔드에 라우트 추가
3. 호출하는 컴포넌트에서 import

### "디자인을 수정하고 싶다"
1. 전체 색상: `tailwind.config.js`
2. 개별 컴포넌트: 해당 컴포넌트 파일의 className
3. 레이아웃: `src/components/Layout.jsx`
```

### 3. 유지보수 가이드 (Maintenance Guide)
**파일명:** `[프로젝트명]_유지보수.md`

바이브코더가 AI 없이도 기본적인 운영/수정을 할 수 있게 하는 가이드.

```markdown
# [프로젝트명] 유지보수 가이드

## 실행 방법
```bash
# 의존성 설치
npm install

# 개발 서버 실행
npm run dev

# 빌드
npm run build
```

## 환경변수 (.env)
| 변수명 | 설명 | 예시 | 필수 |
|--------|------|------|------|
| DATABASE_URL | DB 연결 주소 | postgresql://... | ✅ |
| JWT_SECRET | 토큰 암호 키 | 아무 긴 문자열 | ✅ |

## 자주 하는 작업

### 텍스트/이미지 변경
→ `src/` 안에서 해당 텍스트 검색 → 수정 → 저장 → 자동 반영

### 새 상품 추가 (이커머스인 경우)
→ 관리자 페이지 → 상품 관리 → 추가 버튼

### 에러가 날 때
1. 브라우저 콘솔(F12) 확인
2. 터미널의 에러 메시지 확인
3. 에러 메시지를 Claude에게 보여주고 수정 요청

## 배포 방법
[프로젝트별 배포 절차]

## 백업
[DB 백업 방법 및 주기]

## 비용
| 서비스 | 무료 범위 | 유료 전환 시점 |
|--------|----------|---------------|
| Vercel | 월 100GB 대역폭 | 트래픽 초과 시 |
| Railway | $5 무료 크레딧/월 | DB 크기 초과 시 |
```

## 작동 방식

### 자동 가동 조건
다음 상황에서 자동으로 문서화를 시작한다:
1. vibe-dev가 개발을 완료했을 때
2. 사용자가 "문서화해줘" / "정리해줘"라고 할 때
3. 프로젝트의 중요 마일스톤이 완료되었을 때

### 문서 생성 프로세스

```
[입력 수집]
├── vibe-planner의 기획서 + 기획 대화 로그
├── vibe-reviewer의 검수 리포트
├── vibe-dev의 코드 + 개발 로그
└── 전체 대화에서 사용된 프롬프트
         ↓
[분석]
├── 핵심 결정사항 추출
├── 프로젝트 구조 분석
├── 파일간 의존성 맵핑
└── 효과적이었던 프롬프트 식별
         ↓
[문서 생성]
├── 프로젝트 레시피 (재사용 참조 문서)
├── 코드 맵 (유지보수용 지도)
└── 유지보수 가이드 (운영 매뉴얼)
         ↓
[사용자에게 전달]
```

### 프롬프트 기록 기준

모든 프롬프트를 다 기록하지 않는다. **효과적이었던 프롬프트**만 선별:

**기록하는 것:**
- 핵심 기능을 만들어낸 프롬프트
- 복잡한 에러를 해결한 프롬프트
- 기획 방향을 결정지은 프롬프트
- "이 프롬프트 덕분에 결과가 좋았다"고 판단되는 것

**기록하지 않는 것:**
- "네" / "좋아" 같은 단순 확인
- 오타 수정 같은 사소한 요청
- 실패한 프롬프트 (단, 실패 원인이 교훈적이면 기록)

### 프롬프트에서 패턴 추출

단순히 프롬프트를 나열하는 것이 아니라, **패턴**을 추출한다:

```markdown
## 효과적인 프롬프트 패턴

### 기능 요청 패턴
"[기능명]을 만들어줘.
- [세부사항 1]
- [세부사항 2]
- [참고할 것: 기존 코드의 XX 패턴을 따라서]"

→ 이 패턴으로 요청하면 기존 코드와 일관된 결과가 나옴

### 디버깅 패턴
"[파일명]의 [라인 근처]에서 [에러 메시지]가 발생.
기대 동작은 [X]인데 실제로는 [Y]가 됨.
[관련 코드 붙여넣기]"

→ 에러 위치 + 기대/실제 + 코드를 함께 주면 수정 정확도 높음
```

## 문서 품질 기준

좋은 바이브코딩 문서란:

1. **바이브코더가 읽고 이해할 수 있어야 한다** — 개발자 전용 문서가 아님
2. **다음 프로젝트에서 실제로 소환해서 쓸 수 있어야 한다** — 레시피의 핵심
3. **"어느 파일을 건드려야 하나?"에 30초 내 답할 수 있어야 한다** — 코드 맵의 핵심
4. **"어떻게 실행/배포하나?"에 1분 내 답할 수 있어야 한다** — 유지보수 가이드의 핵심
