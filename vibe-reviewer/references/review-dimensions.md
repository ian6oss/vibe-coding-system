# 검수 차원별 상세 기준

검수팀의 3명의 검수관이 각각 어떤 기준으로 기획서를 평가하는지의 상세 가이드.

---

## 기술 검수관 (Tech Reviewer) 상세 기준

### 기술 스택 적합성 평가

**평가 매트릭스:**

| 판단 항목 | 통과 기준 | 미통과 예시 |
|-----------|----------|------------|
| 프론트엔드 선택 | 프로젝트 규모와 팀 역량에 적합 | 간단한 랜딩페이지에 Next.js + GraphQL |
| 백엔드 선택 | 기능 요구사항 충족 가능 | 실시간 채팅에 REST API만 사용 |
| DB 선택 | 데이터 특성에 적합 | 관계형 데이터에 MongoDB 선택 |
| 호스팅 선택 | 트래픽/예산에 적합 | 개인 블로그에 AWS EKS |

### 데이터 모델 검증

확인사항:
- 모든 엔티티(사용자, 게시물, 주문 등)가 정의되었는가?
- 엔티티 간 관계가 명확한가? (1:1, 1:N, N:M)
- 필수 필드와 선택 필드가 구분되었는가?
- 인덱싱이 필요한 필드가 파악되었는가?

### API 설계 건전성

확인사항:
- 모든 기능에 대응하는 API 엔드포인트가 있는가?
- 인증/인가 방식이 정의되었는가?
- 에러 응답 형식이 통일되었는가?
- Rate limiting 고려가 있는가?

### 보안 체크포인트

**필수 보안 항목:**
- HTTPS 적용
- 비밀번호 해싱 방식
- SQL Injection 방지
- XSS 방지
- CSRF 방지
- API 키/시크릿 관리 방법

**결제가 있는 경우 추가:**
- PCI DSS 준수 방법
- 결제 데이터 처리 범위
- 결제 대행사 연동 방식

## UX 검수관 (UX Reviewer) 상세 기준

### 사용자 흐름 완전성 검증

**핵심 플로우 체크:**
각 핵심 기능에 대해 아래를 확인:
1. 진입점이 명확한가? (사용자가 이 기능을 어떻게 찾는가)
2. 단계가 논리적인가? (불필요한 단계는 없는가)
3. 완료 확인이 있는가? (성공 메시지, 확인 페이지)
4. 되돌리기가 가능한가? (실수했을 때)
5. 이탈 시 어떻게 되는가? (중간에 나가면 데이터가 보존되는가)

### 화면 상태 매트릭스

모든 주요 화면에 대해 다음 상태가 정의되었는지 확인:

| 화면 | 로딩 중 | 데이터 있음 | 데이터 없음 | 에러 | 권한 없음 |
|------|---------|-----------|-----------|------|----------|
| 목록 | ?       | ?         | ?         | ?    | ?        |
| 상세 | ?       | ?         | -         | ?    | ?        |
| 폼   | ?       | ?         | -         | ?    | ?        |

### 인터랙션 정의 수준

**기본 (반드시):**
- 버튼 클릭 시 동작
- 폼 제출 시 동작
- 링크 이동 목적지

**중급 (권장):**
- 호버 상태
- 포커스 상태
- 비활성화 상태
- 터치/스와이프 동작 (모바일)

**고급 (선택):**
- 마이크로 인터랙션
- 페이지 전환 애니메이션
- 스크롤 기반 동작

### 접근성 기본 체크

- 이미지에 대체 텍스트가 고려되었는가?
- 색상만으로 정보를 전달하지 않는가?
- 폼 요소에 라벨이 있는가?
- 키보드만으로 모든 기능을 사용할 수 있는가?

## 비즈니스 검수관 (Business Reviewer) 상세 기준

### 수익 모델 정합성

| 수익 모델 | 확인 항목 |
|-----------|----------|
| 광고 기반 | 광고 위치 정의, 광고 네트워크 선택, 사용자 경험 영향 |
| 구독 모델 | 티어 정의, 기능 제한 방법, 결제 주기, 취소 정책 |
| 거래 수수료 | 수수료율, 정산 주기, 분쟁 해결 절차 |
| 일회성 판매 | 가격 정책, 환불 정책, 라이선스 조건 |
| 프리미엄 | 무료/유료 기능 구분, 전환 유도 전략 |

### 법적 요구사항 체크

**한국 기준 필수:**
- 개인정보처리방침 (개인정보보호법)
- 이용약관
- 만 14세 미만 가입 제한 또는 법정대리인 동의
- 전자상거래법 준수 (이커머스의 경우)
- 통신판매업 신고 (해당 시)

**글로벌 서비스 추가:**
- GDPR (EU 사용자)
- CCPA (캘리포니아 사용자)
- Cookie Consent

### MVP 범위 평가

**적절한 MVP란:**
- 핵심 가치를 검증할 수 있는 최소 기능 세트
- 보통 전체 기획의 30-40%에 해당
- 2-4주 내 개발 가능한 수준

**MVP가 너무 클 때의 신호:**
- 기능이 10개 이상
- "다 중요해요"라는 답변
- 3rd party 연동이 3개 이상

**MVP가 너무 작을 때의 신호:**
- 핵심 기능이 빠져있음
- 사용자가 가치를 느끼기 어려움
- "이걸로 뭘 할 수 있지?" 반응이 예상됨

### 운영 부담 평가

기획서에 있는 기능들의 운영 비용을 추정:

| 기능 | 운영 부담 | 주의 사항 |
|------|----------|----------|
| 사용자 생성 콘텐츠 | 높음 | 모더레이션 필요 |
| 실시간 채팅 | 높음 | 서버 비용 + 모니터링 |
| 이메일 발송 | 중간 | 발송 비용 + 스팸 관리 |
| 파일 업로드 | 중간 | 스토리지 비용 |
| 정적 페이지 | 낮음 | 거의 무관리 |

## 종합 등급 산정 기준

### A등급 (바로 개발 가능)
- 치명적 이슈: 0개
- 중요 이슈: 0-2개 (경미한 수준)
- 기술 스택 완전 정의
- 모든 핵심 기능의 세부사항 존재
- 데이터 모델 정의됨
- 사용자 흐름 완전함

### B등급 (소소한 보완 후 개발 가능)
- 치명적 이슈: 0개
- 중요 이슈: 3-5개
- 기본 기술 방향 설정됨
- 핵심 기능 대부분 정의됨
- 일부 엣지 케이스 미정의

### C등급 (주요 사항 보완 필요)
- 치명적 이슈: 1-2개
- 중요 이슈: 5개 이상
- 기술 스택 일부 미정
- 주요 기능 일부 세부사항 부족

### D등급 (상당 부분 재작업 필요)
- 치명적 이슈: 3개 이상
- 주요 기능의 세부사항 대부분 부족
- 사용자 흐름에 큰 빈 곳

### F등급 (기획 재시작 권장)
- 프로젝트의 본질이 불명확
- 타겟 사용자가 정의되지 않음
- 핵심 기능조차 리스트화되지 않음
